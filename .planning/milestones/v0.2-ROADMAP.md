# Roadmap: efx-skills v0.2.0 — Origin & Metadata

## Overview

v0.2.0 transforms efx-skills from a stateless installer into a tool that knows the provenance of every skill. The work flows from a metadata foundation in config.json, through display and integration touchpoints (search, config page, browser), into active capabilities (update, doctor) that leverage that metadata. Each phase delivers a complete, verifiable capability building toward the core value: every installed skill knows where it came from.

## Phases

**Phase Numbering:**
- Integer phases (1, 2, 3): Planned milestone work
- Decimal phases (2.1, 2.2): Urgent insertions (marked with INSERTED)

Decimal phases appear between their surrounding integers in numeric order.

- [ ] **Phase 1: Config Metadata Schema** - Extend config.json with skills array, skills-path, and repo URLs
- [ ] **Phase 2: Origin Tracking** - Display origin in search, write/remove metadata on install/uninstall
- [ ] **Phase 3: Config Page Redesign** - Cleaner registry/repo presentation and label rename
- [ ] **Phase 4: Browser Integration** - Open skill/registry URLs in browser from any view
- [ ] **Phase 5: Update System** - Verify, update, and globally update skills from upstream
- [x] **Phase 6: Doctor** - Verify config-filesystem consistency and backfill legacy metadata (completed 2026-03-05)

## Phase Details

### Phase 1: Config Metadata Schema
**Goal**: Config.json has the structural foundation to store full skill provenance
**Depends on**: Nothing (first phase)
**Requirements**: ORIG-02, ORIG-03, ORIG-04
**Success Criteria** (what must be TRUE):
  1. Config.json contains a `skills` array capable of holding per-skill metadata (owner, name, registry type, URL)
  2. Config.json contains a `skills-path` field pointing to the central skill storage directory
  3. Each repo entry in config.json includes a `url` field with the GitHub URL
  4. Existing config files load without error (backward compatible -- missing fields get defaults)
**Plans:** 1 plan

Plans:
- [ ] 01-01-PLAN.md — Extend ConfigData schema and parameterize Store with skills-path

### Phase 2: Origin Tracking
**Goal**: Users see where skills come from, and install/uninstall keeps metadata in sync
**Depends on**: Phase 1
**Requirements**: ORIG-01, ORIG-05, ORIG-06
**Success Criteria** (what must be TRUE):
  1. Search results display the origin (owner/repo) next to each skill name
  2. Installing a skill writes its full metadata (owner, name, registry, URL) to the config.json skills array
  3. Uninstalling or removing a skill removes its metadata entry from the config.json skills array
**Plans:** 4 plans

Plans:
- [ ] 02-01-PLAN.md — TDD: Config mutation functions (saveConfigData, addSkillToConfig, removeSkillFromConfig, skillMetaFromAPISkill)
- [ ] 02-02-PLAN.md — Wire metadata functions into install and removal flows
- [ ] 02-03-PLAN.md — Gap closure: Registry column in search, version/timestamp on install, manage help bar UX
- [ ] 02-04-PLAN.md — Gap closure: Separate [t] toggle (symlink-only) from [r] remove (full deletion with confirmation)

### Phase 3: Config Page Redesign
**Goal**: Config page presents registries and repos in a clean, scannable format
**Depends on**: Phase 1
**Requirements**: CONF-01, CONF-02, CONF-03
**Success Criteria** (what must be TRUE):
  1. Config page shows registries with friendly display names (e.g., "Vercel" not "skills.sh") and bold URLs
  2. Custom repos display as two-column `owner    repo-name` format
  3. The section formerly labeled "Providers" now reads "Providers search"
**Plans:** 1 plan

Plans:
- [ ] 03-01-PLAN.md — TDD: Config page display redesign (friendly registry names, two-column repos, providers label)

### Phase 4: Browser Integration
**Goal**: Users can open any skill or registry source URL directly in their browser
**Depends on**: Phase 2
**Requirements**: BRWS-01, BRWS-02, BRWS-03, BRWS-04, BRWS-05
**Success Criteria** (what must be TRUE):
  1. Pressing [o] in search view opens the selected skill's source URL in the default browser
  2. Pressing [o] in manage/provider view opens the skill or group URL in the browser
  3. Pressing [o] in config view opens the registry or repo GitHub URL in the browser
  4. Browser open works cross-platform using `open` (macOS) and `xdg-open` (Linux)
  5. Playbooks.com skills open a skill-specific URL when available, falling back to the playbooks.com domain
**Plans:** 2 plans

Plans:
- [ ] 04-01-PLAN.md — TDD: Browser utility (openInBrowser, urlForAPISkill, registryBaseURL, urlForManagedSkill)
- [ ] 04-02-PLAN.md — Wire [o] keybinding into search, manage, and config views

### Phase 5: Update System
**Goal**: Users can check for and apply upstream changes to installed skills
**Depends on**: Phase 2
**Requirements**: UPDT-01, UPDT-02, UPDT-03, UPDT-04
**Success Criteria** (what must be TRUE):
  1. Pressing [v] on a skill checks upstream for newer commits and reports whether an update is available
  2. Pressing [u] on a skill re-downloads it from upstream, replacing the local copy
  3. Pressing [g] triggers a global update of all installed skills
  4. The installed git commit hash is stored in config/lock so subsequent verify checks compare against it
**Plans:** 2 plans

Plans:
- [ ] 05-01-PLAN.md — TDD: Update system core (commit hash fetch, storage, comparison, update functions)
- [ ] 05-02-PLAN.md — Wire [v], [u], [g] keybindings into manage view

### Phase 6: Doctor
**Goal**: Users can verify and repair the integrity of their skill installation
**Depends on**: Phase 2
**Requirements**: DCTR-01, DCTR-02, DCTR-03, DCTR-04
**Success Criteria** (what must be TRUE):
  1. Running doctor reports any skills tracked in config that are missing from the filesystem
  2. Running doctor detects skills present on filesystem but not tracked in config, and offers to add them
  3. Running doctor backfills metadata for pre-v0.2.0 skills using the known correspondence list
  4. Doctor prints clear alert messages for every problem found, with actionable guidance
