<feature>
  <name>Browser utility: openInBrowser + URL resolvers</name>
  <files>internal/tui/browser.go, internal/tui/browser_test.go</files>
  <behavior>
    ## openInBrowser(url string) error
    - On darwin: executes `open {url}` via exec.Command, using cmd.Start() (non-blocking)
    - On linux: executes `xdg-open {url}` via exec.Command, using cmd.Start() (non-blocking)
    - On unsupported OS: returns error "unsupported platform: {os}"
    - NOTE: Do NOT test actual browser launch -- test the command construction. Use a test helper or accept that this function is a thin wrapper. Focus tests on URL construction.

    ## urlForAPISkill(s api.Skill) string
    - skills.sh registry (or any non-playbooks): returns `https://github.com/{Source}` when Source is non-empty
    - skills.sh with empty Source: returns ""
    - playbooks.com registry: returns `https://playbooks.com/skills/{Source}/{Name}` when both Source and Name are non-empty
    - playbooks.com with empty Source or Name: returns `https://playbooks.com` (fallback)

    ## registryBaseURL(name string) string
    - "skills.sh" -> "https://skills.sh"
    - "playbooks.com" -> "https://playbooks.com"
    - unknown -> "" (empty string)

    ## urlForManagedSkill(skillName string, skills []SkillMeta) string
    - When a SkillMeta with matching Name exists: returns its URL field
    - When no match: returns ""
    - When skills slice is nil: returns ""

    Test cases (input -> expected):
    - urlForAPISkill({Source:"vercel-labs/ai-chatbot-skills", Name:"web-design-guidelines", Registry:"playbooks.com"}) -> "https://playbooks.com/skills/vercel-labs/ai-chatbot-skills/web-design-guidelines"
    - urlForAPISkill({Source:"acme/tools", Name:"lint", Registry:"skills.sh"}) -> "https://github.com/acme/tools"
    - urlForAPISkill({Source:"", Name:"lint", Registry:"skills.sh"}) -> ""
    - urlForAPISkill({Source:"", Name:"", Registry:"playbooks.com"}) -> "https://playbooks.com"
    - registryBaseURL("skills.sh") -> "https://skills.sh"
    - registryBaseURL("playbooks.com") -> "https://playbooks.com"
    - registryBaseURL("unknown") -> ""
    - urlForManagedSkill("lint", []SkillMeta{{Name:"lint", URL:"https://github.com/acme/tools"}}) -> "https://github.com/acme/tools"
    - urlForManagedSkill("missing", []SkillMeta{{Name:"lint", URL:"https://github.com/acme/tools"}}) -> ""
    - urlForManagedSkill("anything", nil) -> ""
  </behavior>
  <implementation>
    Create `internal/tui/browser.go` in package `tui` with:

    1. `openInBrowser(url string) error` -- Uses `runtime.GOOS` switch to dispatch:
       - "darwin": `exec.Command("open", url).Start()`
       - "linux": `exec.Command("xdg-open", url).Start()`
       - default: `fmt.Errorf("unsupported platform: %s", runtime.GOOS)`
       CRITICAL: Use `cmd.Start()` NOT `cmd.Run()` -- the TUI must not block.

    2. `urlForAPISkill(s api.Skill) string` -- URL resolver for search view skills:
       - Switch on `s.Registry`:
         - "playbooks.com": if Source and Name non-empty, return `https://playbooks.com/skills/{Source}/{Name}`. Else return `https://playbooks.com`.
         - default (skills.sh, github, etc.): if Source non-empty, return `https://github.com/{Source}`. Else return "".

    3. `registryBaseURL(name string) string` -- Maps registry API name to browser-friendly base URL:
       - "skills.sh" -> "https://skills.sh"
       - "playbooks.com" -> "https://playbooks.com"
       - default -> ""

    4. `urlForManagedSkill(skillName string, skills []SkillMeta) string` -- Looks up URL from config skills array:
       - Iterate skills, return first match where `s.Name == skillName`.
       - Return "" if no match or skills is nil.
       NOTE: This takes a `[]SkillMeta` parameter (not reading config from disk) so it's testable without filesystem. The caller in manage.go will load config and pass `cfg.Skills`.

    Imports needed: `fmt`, `os/exec`, `runtime`, `github.com/lmarques/efx-skills/internal/api`
  </implementation>
</feature>
