<feature>
  <name>Feature: Doctor diagnostic engine (config-filesystem consistency, backfill, reporting)</name>
  <files>internal/tui/doctor.go, internal/tui/doctor_test.go</files>
  <behavior>
    Phase 1 - Types and known correspondences:
    - DoctorIssue struct with fields: Severity (string: "error", "warning", "info"), Category (string: "missing", "untracked", "backfill"), SkillName (string), Message (string), Fix (string)
    - DoctorReport struct with fields: Issues ([]DoctorIssue), MissingFromFS ([]string), UntrackedOnFS ([]string), BackfillCandidates ([]string)
    - knownCorrespondences is a package-level var: map[string]SkillMeta mapping skill directory names to their known metadata. This covers pre-v0.2.0 skills that were installed before origin tracking existed. Populate with skills from the default registries and repos in config.go (skills.sh, playbooks.com repos). The map should contain at least the skills from the default repos: yoanbernabeu/grepai-skills, better-auth/skills, awni/mlx-skills.

    Phase 2 - RunDiagnostics:
    - RunDiagnostics(skillsPath string, configSkills []SkillMeta) (*DoctorReport, error)
    - Lists directories in skillsPath (each directory = an installed skill)
    - Compares against configSkills by Name field
    - Skills in config but NOT on filesystem -> MissingFromFS + error-severity issue
    - Skills on filesystem but NOT in config -> UntrackedOnFS + warning-severity issue
    - Untracked skills that exist in knownCorrespondences -> also added to BackfillCandidates + info-severity issue
    - Returns structured report with all issues

    Phase 3 - BackfillLegacySkills:
    - BackfillLegacySkills(candidates []string) (backfilled []string, err error)
    - For each candidate skill name, looks up knownCorrespondences
    - Calls addSkillToConfig with the known metadata
    - Returns list of successfully backfilled skill names
    - Gracefully handles individual failures (continues with next)

    Phase 4 - FormatReport:
    - FormatReport(report *DoctorReport) string
    - Formats issues as clear alert messages with severity prefix
    - Groups by category: errors first, then warnings, then info
    - Each issue shows: severity icon, skill name, message, and suggested fix
    - Summary line at end: "N issues found (X errors, Y warnings, Z info)" or "No issues found - all skills healthy"
  </behavior>
  <implementation>
    1. Create internal/tui/doctor.go with package tui

    2. Define DoctorIssue and DoctorReport structs

    3. Define knownCorrespondences map. Populate it with skills from the default repos. Since we don't have a definitive list at compile time, use a reasonable set. The key is the skill directory name (e.g., "grepai"), and the value is a SkillMeta with Owner (e.g., "yoanbernabeu/grepai-skills"), Name, Registry ("github"), and URL. At minimum, include entries for skills from the three default repos. The exact skill names should be discovered by checking if they match lock file entries.

    IMPORTANT: The knownCorrespondences map is a best-effort lookup. The primary backfill mechanism works like this:
    - First, check the lock file for the skill. The lock file's LockEntry has Source (owner/repo format) and SourceType fields that provide all the metadata needed.
    - If the skill IS in the lock file but NOT in config, construct SkillMeta from lock data: Owner=Source, Name=skillName, Registry=SourceType (or "github"), URL=SourceURL.
    - If the skill is NOT in the lock file either, fall back to knownCorrespondences.
    - This means the lock file is the primary data source for backfill, not the hardcoded map.

    4. Implement RunDiagnostics:
       - Read filesystem: os.ReadDir(skillsPath) to get installed skill directories
       - Build sets: configSet (from configSkills by Name), fsSet (from directory names)
       - Compare sets to find MissingFromFS and UntrackedOnFS
       - For UntrackedOnFS, check lock file first, then knownCorrespondences for backfill candidates
       - Build issues list with appropriate severity, message, and fix

    5. Implement BackfillLegacySkills:
       - Accept candidates list and a skillsPath string
       - For each candidate, look up lock file first (via Store.ReadLockFile), then knownCorrespondences
       - Construct SkillMeta and call addSkillToConfig
       - Collect successes and continue on failures

    6. Implement FormatReport:
       - Use severity icons: "!" for error, "?" for warning, "i" for info
       - Format each issue on its own line
       - Group by severity (errors first)
       - End with summary line

    7. Write comprehensive tests in doctor_test.go:
       - TestRunDiagnostics_MissingFromFS: config has skill X, filesystem doesn't -> error issue
       - TestRunDiagnostics_UntrackedOnFS: filesystem has skill Y, config doesn't -> warning issue
       - TestRunDiagnostics_BackfillCandidate: untracked skill matches lock file entry -> info + backfill candidate
       - TestRunDiagnostics_AllHealthy: config and filesystem match -> no issues
       - TestBackfillLegacySkills: candidates get added to config via addSkillToConfig
       - TestFormatReport_WithIssues: formatted output contains all issues grouped correctly
       - TestFormatReport_Healthy: formatted output shows "No issues found"

    Tests use t.TempDir() for filesystem setup. For config functions, tests need to:
    - Set HOME env var to temp dir
    - Create config.json and skills directories in the temp structure
    - Tests that call addSkillToConfig need HOME set so loadConfigFromFile reads from temp

    NOTE: doctor.go lives in package tui because it needs access to unexported functions (loadConfigFromFile, addSkillToConfig, defaultSkillsPath). This is consistent with how other features (browser, manage, config) are organized in this codebase.
  </implementation>
</feature>
