<feature>
  <name>Feature: Update system core (commit hash fetch, store, compare, update)</name>
  <files>internal/skill/store.go, internal/skill/store_test.go, internal/tui/search.go</files>
  <behavior>
    Phase 1 - CommitHash field and storage:
    - LockEntry gains a `CommitHash` field (json:"commitHash")
    - AddToLock accepts an optional commitHash parameter and stores it in the lock entry
    - Existing lock files without commitHash load correctly (backward compatible, empty string default)

    Phase 2 - FetchLatestCommitHash:
    - FetchLatestCommitHash(owner, repo string) (string, error) calls GitHub API
    - GET https://api.github.com/repos/{owner}/{repo}/commits?per_page=1 returns SHA
    - Returns the 40-char hex SHA string from the response
    - Returns error on HTTP failure or non-200 status
    - Test uses httptest.NewServer to mock the GitHub API response

    Phase 3 - CheckForUpdate:
    - CheckForUpdate(skillName string) (hasUpdate bool, currentHash string, latestHash string, err error)
    - Reads lock file to get stored commitHash and source for the skill
    - Calls FetchLatestCommitHash with parsed owner/repo from source
    - Returns hasUpdate=true when hashes differ, false when they match
    - Returns error when skill not found in lock file
    - Returns hasUpdate=true when stored commitHash is empty (legacy install, always treat as updateable)

    Phase 4 - UpdateSkill:
    - UpdateSkill(skillName string) error
    - Reads lock file to get source for the skill
    - Calls Install(source, skillName) to re-download
    - Fetches latest commit hash and updates lock entry with new hash and UpdatedAt timestamp
    - Returns error when skill not found in lock file

    Phase 5 - UpdateAllSkills:
    - UpdateAllSkills() (updated []string, err error)
    - Reads lock file, iterates all skills
    - For each skill, calls CheckForUpdate; if hasUpdate, calls UpdateSkill
    - Returns list of skill names that were updated
    - Gracefully handles individual skill update failures (continues with next, collects errors)
  </behavior>
  <implementation>
    1. Add CommitHash field to LockEntry struct (json:"commitHash")
    2. Modify AddToLock to accept and store commitHash -- change signature to AddToLock(skillName, source, commitHash string)
    3. Add FetchLatestCommitHash function using net/http GET to api.github.com, parsing JSON response for sha field. Use a struct-level or package-level variable for the base URL so tests can override it.
    4. Add CheckForUpdate that reads lock, parses source into owner/repo, fetches latest hash, compares
    5. Add UpdateSkill that re-installs and updates lock entry
    6. Add UpdateAllSkills that iterates lock and updates each skill with available updates
    7. Update the install flow's AddToLock call site in internal/tui/search.go to pass the commit hash (fetch it after install, pass to AddToLock). This is a one-line change in search.go's installStartMsg handler.

    IMPORTANT: The GitHub API base URL must be overridable for testing. Use a package-level var:
    ```go
    var gitHubAPIBaseURL = "https://api.github.com"
    ```
    Tests override this to point to httptest.NewServer.

    For FetchLatestCommitHash, parse the GitHub API response which returns an array of commit objects. Each has a "sha" field at the top level:
    ```json
    [{"sha": "abc123...", "commit": {...}}]
    ```

    For the AddToLock signature change: update the existing caller in search.go from:
    ```go
    _ = store.AddToLock(s.Name, s.Source)
    ```
    to:
    ```go
    commitHash, _ := skill.FetchLatestCommitHash(owner, repo)
    _ = store.AddToLock(s.Name, s.Source, commitHash)
    ```
    where owner/repo is parsed from s.Source (split on "/").

    NOTE: This plan spans 3 packages (internal/skill for core logic + tests, internal/tui for call site update). The cross-package change in search.go is a single call site update to match the new 3-arg AddToLock signature -- it must happen in this plan to keep the build green.
  </implementation>
</feature>
